Playing with kubeadm in Vagrant Machines, Part 2

Ever since I used kubeadm to create Kubernetes in Vagrant boxes, it’s been pretty cool to bootstrap my own mini-cluster on my local machine but I only explored the tip of the iceberg. kubeadm is not without its quirks, after all. I like it for basic learning but I really don’t recommend you try to use it to build a very complex cluster (yet, since at the time of this post, it’s still under development). I got kubeadm to create my cluster and thanks to a reader, figured out a problem with rebooting the existing cluster. I proceeded to play with Kubernetes configurations in various ways, including:

  • Service Meshes
  • RBAC
  • Deployment of Services

I did it enough times that I ended up writing some scripts to help me run kubeadm init on the head and collect the token. Basically, after the boxes come up, I run the following on the head node.

# On the head node
$ /tmp/init-head.sh
$ cat /tmp/join.sh # get the token

Everything was great — I could deploy a service like nginx and proxy to it. However, I hit a snag when I wanted to use busybox to debug something with an application container. For some reason, running kubectl exec appeared with an error message!

Hmmm…HTTP Error Code 404 all around. Something seemed suspicious, as if it couldn’t properly find where the logs or container was. How exactly does kubectl exec and kubectl logs work?

Start with an Educated Guess

With the 404 Error Code, I wondered if the connection with the kubelet is wrong. Using a handy-dandy search engine, I stumbled upon a simple explanation on how kubectl exec works. The key step that caught my attention was that the API server makes a connection with the worker’s kubelet to pass the command for execution. Tracing through some of the Kubernetes code, I found the SPDY negotiation step between the API server and the kubelet and the round_trippers.go code the debug information references with the error. It seems like the right server endpoint needs to be passed in order for the connection to happen. As explained by Kubernetes’s page on Master-Node connection, the API server communicates with the kubelet based on its HTTPS endpoint. What is the kubelet’s HTTP endpoint on my worker1 VM?

$ kubectl get nodes worker1 -o yaml
apiVersion: v1
kind: Node
...
status:
  addresses:
  - address: 10.0.2.15
...

Wait a minute, that isn’t the right IP address for worker1! I recalled the a similar problem when I ran kubeadm for the master:

  • Vagrant creates two network interfaces for each machine. eth0 is NAT networked, eth1 is a private network.
  • The main Kubernetes interface is on eth1.
  • In order for the worker to properly join the master, I had to explicitly state the --apiserver-advertise-address to point to eth1.

I needed to add an explicit IP address that uses the eth1 interface on the worker, enabling the master’s API server to properly access the worker’s kubelet. I looked at the man page of the kubelet and I found the following option:

--node-ip string    IP address of the node. If set, kubelet will use this IP address for the node

Looks like I need to set it in the kubelet — but how?

Changing the kubelet Configuration

Examining some documentation, I found that kubeadm cannot inject anything into the kubelet’s drop-in configuration via CLI. Instead, I had to edit the kubelet’s configuration file to add the additional argument.

$ less /etc/systemd/system/kubelet.service.d/10-kubeadm.conf
[Service]
Environment="KUBELET_KUBECONFIG_ARGS=--bootstrap-kubeconfig=/etc/kubernetes/bootstrap-kubelet.conf --kubeconfig=/etc/kubernetes/kubelet.conf"
Environment="KUBELET_SYSTEM_PODS_ARGS=--pod-manifest-path=/etc/kubernetes/manifests --allow-privileged=true"
Environment="KUBELET_NETWORK_ARGS=--network-plugin=cni --cni-conf-dir=/etc/cni/net.d --cni-bin-dir=/opt/cni/bin"
Environment="KUBELET_DNS_ARGS=--cluster-dns=10.96.0.10 --cluster-domain=cluster.local"
Environment="KUBELET_AUTHZ_ARGS=--authorization-mode=Webhook --client-ca-file=/etc/kubernetes/pki/ca.crt"
Environment="KUBELET_CADVISOR_ARGS=--cadvisor-port=0"
Environment="KUBELET_CGROUP_ARGS=--cgroup-driver=systemd"
Environment="KUBELET_CERTIFICATE_ARGS=--rotate-certificates=true --cert-dir=/var/lib/kubelet/pki"
Environment="KUBELET_EXTRA_ARGS=--node-ip=<worker IP address>"
ExecStart=
ExecStart=/usr/bin/kubelet $KUBELET_KUBECONFIG_ARGS $KUBELET_SYSTEM_PODS_ARGS $KUBELET_NETWORK_ARGS $KUBELET_DNS_ARGS $KUBELET_AUTHZ_ARGS $KUBELET_CADVISOR_ARGS $KUBELET_CGROUP_ARGS $KUBELET_CERTIFICATE_ARGS $KUBELET_EXTRA_ARGS

Then, I restarted the kubelet.

$ systemctl daemon-reload
$ systemctl restart kubelet

There other methods of updating the kubelet configuration outlined here but they are still alpha and I wasn’t making updates to a cluster with significant usage.

Putting it All Together

It stands to reason that if the kubelet configuration isn’t correct on the worker, I need to correct it for the head node, too. I updated my Vagrantfile to upload a script into the head and worker for me. For the worker, the script runs the kubeadm join command to add the worker to the cluster, updates the kubelet configuration with the correct IP address, and reloads the kubelet. On the head node, it also corrects the IP address and reloads the kubelet. To create my Kubernetes cluster, I run the following:

  • From my laptop, I run vagrant up and vagrant ssh to the head node to run my init script.
  • On my head node, I run the init-head.sh script to create my Kubernetes head node.
# On the head node, run the init script with the pod CIDR
$ /tmp/init-head.sh 10.1.0.0/16
configmap "kube-router-cfg" created
daemonset "kube-router" created
serviceaccount "kube-router" created
clusterrole "kube-router" created
clusterrolebinding "kube-router" created
  • My head node has a file with the bootstrap token I can use to run on the worker node.
$ less /tmp/join.sh
kubeadm join --token REDACTED 192.168.205.10:6443 --discovery-token-ca-cert-hash REDACTED
  • I use vagrant ssh to log into my worker1 node and pass the token I’ve generated to the init-worker.sh script.
$ /tmp/init-worker.sh "kubeadm join --token REDACTED 192.168.205.10:6443 --discovery-token-ca-cert-hash REDACTED...This node has joined the cluster:
* Certificate signing request was sent to master and a response
  was received.
* The Kubelet was informed of the new secure connection details.Run 'kubectl get nodes' on the master to see this node join the cluster.
  • Now, when I check my head or worker1 node for the IP address, I see it set as the node’s address. (Below, I use jq to grab the right JSON key from worker1.)
$ kubectl get nodes worker1 -o json | jq .status.addresses
[
  {
    "address": "192.168.205.11",
    "type": "InternalIP"
  },
  {
    "address": "worker1",
    "type": "Hostname"
  }
]

When I try to exec into my busybox container now, I can successfully get into it!

$ kubectl apply -f busybox.yml
deployment "busybox" created
$ kubectl get pods
NAME                       READY     STATUS    RESTARTS   AGE
busybox-7c9687585b-8s2fx   1/1       Running   0          32s
$ kubectl exec -it busybox-7c9687585b-8s2fx -- /bin/sh
/ #

Summary

While kubeadm isn’t quite ready for bootstrapping production Kubernetes clusters, the problems are super helpful for learning Kubernetes internals and adding the configurations. As I run into some limitations or trouble with default configurations, I’ll be sure to outline them further.
